The lifecycle

Materialize once, reuse while warm.

Materializing a persona is the cost: assembling its playbook, domain knowledge, and skills into one equipped overlay. The warm cache pays that cost once and lets every agent that needs the same persona skip it, until the persona ages out.

01

materialize

Assemble the persona into one equipped overlay: playbook, domain knowledge, skills, worked examples.

02

execute

The agent performs the task wearing the persona, holding its work to that verification bar.

03

dematerialize

The persona is released when the task completes. The agent is a generalist again.

04

warm

The materialized persona lingers in a shared cache so the next agent reuses it for free.

05

age out

After 30 minutes idle, or a 2 hour ceiling, it is evicted and re-materialized on demand.

What a persona is

A role the agent wears, not a prompt it pastes.

A generalist agent is competent at everything and excellent at nothing. A persona makes an agent excellent at one thing for the length of one task, then steps out of the way.

The role

A domain and a trigger

Each persona owns a territory, backend or security or data, and declares when to equip. The engine scores a task and equips the best fit, the same way skills auto-fire.

The standard

A playbook and a bar

Every persona carries a principal-engineer playbook and a verification bar it holds its own work to before handing off. Equipping one raises the floor for that task.

The source

One file, no magic

The roster lives in personas.toml. Adding a persona is adding a block. The match is curated keywords plus domain vocabulary, with a deterministic tie-break.

How the match works The equip engine scores a task against each persona's curated keywords, where a multi-word phrase counts for more, plus its domain vocabulary, then returns the winner, its score, the terms that fired, and the runners-up. No model call, no network, every decision explainable. With no signal at all it returns the default persona. Run personas match "your task" to preview any decision.

The roster

Ten principal engineers, one muscle memory.

Each is an opinionated specialist with its own domain, playbook, and verification bar. Backend is lit as the default, equipped when a task matches nothing else.

Composes, does not compete

Tier, provider, role.

personas is one of three dimensions of a dispatch. It picks the role and leaves the rest to the tools built for them, so a dispatch is a tier, a provider and model, and a persona.

Tier

model-advisor

Picks the cheapest model tier that credibly preserves quality for the task shape.

Provider / model

provider-forge

Picks the concrete provider and model the work actually runs on.

Role

personas

Picks the principal-engineer persona the agent equips for the task in front of it.

The pack

A gas-city pack your agents get for free.

personas ships as a stdlib-only gas-city pack: the registry, the equip and match engine, the warm-cache lifecycle, an equip-on-task-pickup hook, and a CLI. Vendor it into a city and every agent selects and equips a persona as it claims work.

Read the pack
personas
# the equip decision for a task (read-only)
personas match "audit the upload handler for SSRF"

# equip it: materialize into the shared warm cache
personas equip --from-bead pers-3f1

# what is warm, and how long until it ages out
personas cache
